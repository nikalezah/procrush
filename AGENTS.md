# ProCrush — инструкции для агентов

## Главная цель репозитория

Основная задача репозитория — практика **микросервисной архитектуры** и инфраструктуры: Kubernetes, Kafka, RabbitMQ, Redis.

Домен ProCrush (опросы, личностные профили, мэтчинг, отклики) задаёт реалистичные сценарии: асинхронные задачи, события, кэширование, выделение сервисов. Описание продуктовой логики — в [README.md](./README.md).

## Приоритеты при планировании и реализации

1. Решения, дающие практику целевого стека (cache, messaging, split services, orchestration).
2. Доменные фичи — когда они обоснованы инфраструктурным сценарием.
3. Не упрощать архитектуру «ради скорости», если это мешает изучению целевых технологий.

Реализация — с планкой production-ready: надёжность, наблюдаемость, безопасность и тесты не упрощаются «ради скорости».

## Что не является целью

- Быстрый вывод продукта на рынок.
- Минимальный scope «только чтобы работало» без инфраструктурной ценности.

## Стиль ответов

Разработчик, работающий с репозиторием, **впервые** знакомится с Kubernetes, Kafka, RabbitMQ и Redis. Формулировки должны быть понятны без предположения опыта в этих технологиях:

- Кратко пояснять **зачем** предлагается решение в контексте задачи, а не только **что** сделать.
- При первом упоминании термина — одно простое предложение, что это и какую роль играет (например: «Redis — in-memory хранилище; здесь кэшируем результаты матчинга, чтобы не ходить в PostgreSQL при каждом запросе»).
- Избегать цепочек аббревиатур и жаргона без расшифровки; если термин неизбежен — дать расшифровку или аналогию.
- При выборе между технологиями — коротко сравнить варианты простым языком (когда уместно).
- Пошаговые инструкции и команды — с пояснением, что делает шаг и что ожидать на выходе.
- Не пропускать «очевидное» для senior DevOps: для читателя это новая область.
